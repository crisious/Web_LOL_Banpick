## 1. 제품 개요

이 제품은 `League of Legends 경기 데이터를 입력받아`, 경기 내용을 요약하고 플레이의 장점과 단점을 분석해 주는 웹페이지다.

MVP 단계에서는 `.rofl 원본 직접 처리`를 목표로 하지 않는다. 대신 `Riot Match/Timeline 기반 데이터` 또는 `구조화된 경기 데이터`를 사용해, 아래 사용자 가치를 빠르게 검증한다.

- 내 게임을 한 번에 이해할 수 있다
- 내가 잘한 점과 못한 점을 구체적으로 알 수 있다
- 다음 게임에서 무엇을 개선해야 하는지 바로 알 수 있다

## 2. 문제 정의

현재 일반 플레이어는 경기 리플레이를 다시 보더라도 아래 문제를 겪는다.

- 전체 게임 흐름을 빠르게 요약하기 어렵다
- 장면은 기억나도 무엇이 핵심 실수였는지 정리하기 어렵다
- 잘한 점보다 실수만 막연히 느끼거나, 반대로 정확한 약점을 놓친다
- 다음 게임에서 바꿔야 할 행동이 명확하지 않다

우리는 이 문제를 `경기 데이터 기반 자동 요약 + 인사이트 제공`으로 해결한다.

## 3. 목표

### 제품 목표

- 사용자가 경기 1판을 `3분 이내`에 복기할 수 있게 한다
- 추상적 피드백이 아니라 `행동 가능한 개선 포인트`를 제공한다
- 웹 기반 MVP로 분석 가치와 UX 가설을 검증한다

### 비즈니스/검증 목표

- 사용자가 분석 결과를 읽고 `다음 게임에도 다시 쓰고 싶다`고 느끼는지 검증
- 어떤 분석 출력이 가장 유용한지 확인
- 향후 고정밀 분석이나 `.rofl 확장`이 실제로 필요한지 판단

## 4. 비목표

MVP에서는 아래를 하지 않는다.

- `.rofl 원본 파일 직접 처리`
- 리플레이 영상 재생
- 시점 제어, 카메라 이동, 장면 캡처
- 팀 전체 10인 심층 전술 분석
- 실시간 경기 분석
- 패치별 원본 파서 대응

## 5. 대상 사용자

### 1차 타겟

- 자신의 솔로랭크/일반게임 경기 복기를 빠르게 하고 싶은 플레이어
- 긴 리플레이를 직접 다 보기보다 핵심 피드백을 원하는 플레이어
- 티어 상승을 목표로 구체적인 개선 포인트가 필요한 플레이어

### 주요 니즈

- `내가 왜 졌는지` 알고 싶다
- `내가 뭘 잘했는지` 알고 싶다
- `다음 판에서 뭘 바꾸면 되는지` 알고 싶다

## 6. 핵심 사용자 시나리오

### 시나리오 A. 경기 데이터로 바로 분석

1. 사용자가 경기 정보를 입력하거나 샘플 경기를 선택한다
2. 시스템이 경기 데이터를 불러와 분석한다
3. 사용자는 경기 요약, 장점/단점, 핵심 이벤트를 확인한다
4. 사용자는 `다음 게임 체크리스트`를 보고 개선 포인트를 이해한다

### 시나리오 B. 이전 결과 복기

1. 사용자가 저장된 분석 결과를 다시 연다
2. 이전 경기에서 반복되는 약점을 확인한다
3. 향후 비교 분석 가능성에 대한 니즈를 검증한다

MVP에서는 A가 핵심이고, B는 선택 기능이다.

## 7. 핵심 가치 제안

우리 제품은 단순 통계 대시보드가 아니라 아래를 제공해야 한다.

- `요약`: 이 경기가 어떤 흐름으로 흘렀는지
- `해석`: 왜 유리/불리해졌는지
- `평가`: 내가 잘한 점과 부족한 점이 무엇인지
- `행동`: 다음 게임에서 바로 바꿀 수 있는 행동은 무엇인지

## 8. MVP 기능 범위

## 필수 기능

### 8.1 경기 입력

- 소환사명 + 경기 식별자 기반 입력 또는 샘플 경기 선택
- 개발 초기에는 `샘플 JSON 데이터 입력`만으로도 가능
- 사용자는 분석 대상 챔피언, 포지션, 경기 결과를 확인할 수 있어야 한다

### 8.2 경기 요약

- 경기 전체 한줄 요약
- 경기 흐름 요약
- 승패에 영향을 준 핵심 전환점 2~4개

### 8.3 장점 분석

- 플레이어가 잘한 점 3개
- 각 장점에 대한 근거 이벤트 또는 지표
- 장점이 경기 흐름에 어떤 영향을 줬는지 설명

### 8.4 단점 분석

- 플레이어가 아쉬웠던 점 3개
- 각 단점에 대한 근거 이벤트 또는 지표
- 추상적인 비난이 아니라 개선 가능한 표현 사용

### 8.5 개선 액션

- 다음 게임에서 실천할 체크리스트 3~5개
- 가능한 한 행동 단위로 제시
- 예: `첫 귀환 전 와드 타이밍 앞당기기`, `사이드 압박 전 적 정글 위치 확인`

### 8.6 시간대별 핵심 이벤트

- 초반 / 중반 / 후반 구간 요약
- 주요 킬, 오브젝트, 한타, 데스, 로밍 등 이벤트 표시
- 각 이벤트가 왜 중요했는지 짧게 설명

### 8.7 분석 근거 표시

- 분석 결과는 가능한 한 이벤트/지표 근거를 함께 보여준다
- “그냥 AI가 한 말”처럼 느껴지지 않게 설계한다

## 선택 기능

### 8.8 결과 저장

- 최근 분석 1~5건 저장
- 다시 열어보기 가능

### 8.9 샘플 경기 체험

- 실제 입력 없이 바로 데모 가능
- 랜딩 페이지 전환율 검증에 유리

## 9. 사용자 경험 원칙

- 입력은 간단해야 한다
- 결과는 `스크롤만 해도 이해되는 구조`여야 한다
- 통계보다 `의미 해석`이 먼저 보여야 한다
- 비난하는 톤 대신 코칭하는 톤을 사용한다
- 모든 인사이트는 가능하면 `근거 이벤트`와 연결한다

## 13. 분석 로직 방향

MVP 분석은 `규칙 기반 + LLM 기반` 혼합 구조를 권장한다.

### 규칙 기반

- 경기 단계 구분
- 기본 지표 계산
- 중요 이벤트 추출
- 승패 전환점 후보 선정

### LLM 기반

- 자연어 경기 요약 생성
- 장점/단점 서술
- 개선 액션 제안
- 근거 이벤트를 사람이 읽기 쉬운 문장으로 해석

### 중요한 원칙

- LLM은 반드시 `구조화된 입력`을 받아야 한다
- 출력도 `구조화된 JSON`으로 받아 UI에 렌더링한다
- 근거 없는 일반론 생성은 최소화한다

## 14. 결과 출력 포맷 요구사항

분석 결과는 최소한 아래 필드를 포함해야 한다.

- overallSummary
- gameFlowSummary
- strengths[]
- weaknesses[]
- actionChecklist[]
- phaseSummaries[]
- keyMoments[]

각 `strength`와 `weakness`는 아래 속성을 가진다.

- title
- description
- evidence
- impact

각 `keyMoment`는 아래 속성을 가진다.

- timestamp
- label
- reason
- impact

## 15. 성공 지표

### 정성 지표

- 사용자가 `요약이 경기와 맞는다`고 느끼는지
- 사용자가 `다음 게임에서 적용할 포인트가 생겼다`고 느끼는지
- 분석이 `너무 뻔한 말`이 아니라 `내 경기 기반`이라고 느끼는지

### 정량 지표

- 결과 페이지 도달률
- 결과 페이지 체류 시간
- 체크리스트 확인 비율
- 샘플 경기 체험 전환율
- 재방문율 또는 재분석 비율

## 16. 리스크

### 리스크 1. 분석이 너무 일반적일 수 있음

대응:
- 근거 이벤트 기반 출력 강제
- 플레이어 포지션/챔피언 맥락 반영

### 리스크 2. 입력 데이터 품질이 불완전할 수 있음

대응:
- 필수 필드 정의
- 누락 데이터 시 보수적 분석

### 리스크 3. 사용자가 기대하는 정밀도와 실제 결과 차이

대응:
- MVP에서는 `요약/피드백 도구`로 포지셔닝
- 고정밀 리플레이 해석은 후속 연구 과제로 분리

## 17. 출시 기준

아래 조건을 만족하면 MVP 공개 가능 상태로 본다.

- 샘플 경기 기준 결과 페이지가 완성되어 있다
- 경기 요약, 장점, 단점, 체크리스트가 모두 표시된다
- 최소 1개의 입력 경로가 정상 동작한다
- 분석 결과에 근거 이벤트가 함께 표시된다
- UI 흐름이 모바일과 데스크톱에서 모두 사용 가능하다
