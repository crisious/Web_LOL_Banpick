## 1. 문서 목적

이 문서는 MVP 구현을 바로 시작할 수 있도록 작업을 기능 단위 티켓으로 분해한 문서다.

원칙은 아래와 같다.

- 작은 단위로 나눈다
- 각 티켓은 완료 조건이 명확해야 한다
- UI, 데이터, 분석 로직을 분리한다
- MVP 범위를 벗어나는 작업은 뒤로 미룬다

## 3. 상세 티켓

## T001 프로젝트 기본 구조 설정

### 목표

웹 MVP를 빠르게 개발할 수 있는 기본 폴더 구조와 초기 설정을 만든다.

### 작업 내용

- 앱 디렉터리 구조 정의
- 기본 페이지 엔트리 생성
- 공용 컴포넌트 디렉터리 생성
- mock 데이터 디렉터리 생성

### 완료 조건

- 랜딩 페이지와 결과 페이지 라우트가 존재한다
- 공통 구조가 정리되어 이후 티켓이 바로 붙을 수 있다

## T002 공통 레이아웃 및 라우팅 구성

### 목표

헤더와 기본 네비게이션을 포함한 공통 레이아웃을 만든다.

### 작업 내용

- 상단 헤더
- 기본 네비게이션
- 페이지 컨테이너 폭 규칙 정의

### 완료 조건

- 주요 페이지가 동일한 레이아웃 아래에서 렌더링된다

## T003 디자인 토큰/기초 스타일 정의

### 목표

MVP 화면 일관성을 위한 최소한의 시각 규칙을 정의한다.

### 작업 내용

- 색상 변수
- 간격 규칙
- 타이포그래피 계층
- 카드/배지/버튼 기본 스타일

### 완료 조건

- 공통 UI 컴포넌트가 일관된 규칙으로 보인다

## T010 랜딩 페이지 구현

### 목표

서비스 소개와 진입 CTA를 포함하는 첫 화면을 구현한다.

### 작업 내용

- 히어로 섹션
- 가치 제안 카피
- 주요 CTA 버튼
- 결과 미리보기 영역

### 완료 조건

- 사용자가 첫 화면에서 서비스 기능과 다음 행동을 이해할 수 있다

## T011 경기 입력 폼 구현

### 목표

분석 진입을 위한 기본 입력 인터페이스를 만든다.

### 작업 내용

- 소환사명 입력
- 경기 식별자 또는 샘플 선택 UI
- 입력 검증
- 제출 버튼

### 완료 조건

- 유효한 입력일 때 결과 페이지 또는 로딩 흐름으로 이동한다

## T012 샘플 경기 진입 플로우 구현

### 목표

실제 데이터 연동 전에도 완전한 데모 흐름을 만든다.

### 작업 내용

- 샘플 경기 버튼
- 샘플 데이터 로드
- 결과 페이지 연결

### 완료 조건

- 입력 없이도 결과 화면까지 자연스럽게 이동 가능하다

## T020 결과 페이지 레이아웃 구현

### 목표

결과 정보를 보여줄 전체 레이아웃 뼈대를 만든다.

### 작업 내용

- 상단 경기 개요 바
- 좌우 또는 상하 섹션 분할
- 모바일 대응 구조

### 완료 조건

- 와이어프레임 기준 주요 섹션이 모두 자리 잡는다

## T021 경기 개요/요약 카드 구현

### 목표

사용자가 경기 내용을 가장 먼저 이해하는 영역을 만든다.

### 작업 내용

- 챔피언/포지션/승패/시간 표시
- 한줄 요약
- 코치 요약 문단
- 핵심 지표 카드

### 완료 조건

- 결과 페이지 상단만 봐도 경기 맥락을 이해할 수 있다

## T022 장점/단점 카드 구현

### 목표

인사이트를 빠르게 스캔할 수 있는 카드 UI를 만든다.

### 작업 내용

- 장점 카드 3종 UI
- 단점 카드 3종 UI
- 근거 텍스트 슬롯
- 영향 또는 개선 힌트 슬롯

### 완료 조건

- 장점/단점 정보가 명확히 구분되어 읽힌다

## T023 체크리스트 섹션 구현

### 목표

사용자가 다음 게임 행동으로 바로 전환할 수 있게 한다.

### 작업 내용

- 체크리스트 리스트 UI
- 우선순위 시각 표시
- 짧은 보조 설명

### 완료 조건

- 3~5개의 행동 아이템이 읽기 쉽게 노출된다

## T024 타임라인/근거 이벤트 섹션 구현

### 목표

인사이트의 근거가 되는 핵심 이벤트를 시간 순으로 보여준다.

### 작업 내용

- 타임라인 리스트
- 타임스탬프 표시
- 이벤트 유형 배지
- 중요도 강조

### 완료 조건

- 사용자가 경기 전개를 시간 순으로 복기할 수 있다

## T030 샘플 경기 JSON 스키마 정의

### 목표

프론트엔드와 분석 로직이 공통으로 사용할 샘플 입력 형식을 고정한다.

### 작업 내용

- matchInfo 정의
- playerStats 정의
- timelineEvents 정의
- 선택 필드/필수 필드 구분

### 완료 조건

- 샘플 데이터 작성을 위한 명확한 스키마가 존재한다

## T031 샘플 경기 데이터 1건 작성

### 목표

전체 데모 플로우에 사용할 현실감 있는 예시 데이터를 만든다.

### 작업 내용

- 경기 메타 정보 작성
- 핵심 이벤트 10~20개 작성
- 플레이어 스탯 입력

### 완료 조건

- 결과 페이지가 샘플 데이터만으로 충분히 설득력 있게 렌더링된다

## T032 분석 결과 JSON 스키마 정의

### 목표

LLM 또는 Mock 응답이 따라야 할 출력 포맷을 고정한다.

### 작업 내용

- overallSummary
- strengths
- weaknesses
- actionChecklist
- phaseSummaries
- keyMoments

### 완료 조건

- 프론트엔드가 구조화 응답을 안정적으로 렌더링할 수 있다

## T033 Mock 분석 응답 생성기 구현

### 목표

실제 AI 연결 전에도 결과 화면을 완성할 수 있도록 mock 응답을 제공한다.

### 작업 내용

- 샘플 입력에 대한 고정 분석 응답 작성
- 개발 모드에서 mock 연결

### 완료 조건

- 실제 분석 API 없이도 MVP 흐름이 시연 가능하다

## T034 분석 API 계약 정의

### 목표

프론트엔드와 분석 백엔드가 맞물릴 인터페이스를 확정한다.

### 작업 내용

- 요청 형식
- 응답 형식
- 실패 응답 형식
- 로딩/리트라이 정책 정의

### 완료 조건

- API 연동 시 재작업이 최소화된다

## T040 로딩 상태 UI 구현

### 목표

분석 생성 중에도 사용자가 진행 상황을 이해하도록 한다.

### 작업 내용

- 단계형 로딩 메시지
- 스켈레톤 또는 진행 표시

### 완료 조건

- 로딩 중 이탈감을 줄일 수 있다

## T041 에러 상태 UI 구현

### 목표

입력 실패 또는 분석 실패 시 사용자를 다시 흐름으로 복귀시킨다.

### 작업 내용

- 에러 메세지 컴포넌트
- 재시도 CTA
- 샘플 체험 대체 CTA

### 완료 조건

- 실패 상황에서도 사용자가 다음 행동을 알 수 있다

## T042 빈 상태 UI 구현

### 목표

히스토리 등 데이터가 비어 있는 상태를 자연스럽게 처리한다.

### 작업 내용

- 빈 상태 일러스트 또는 안내 영역
- 첫 분석 시작 CTA

### 완료 조건

- 빈 페이지가 아니라 안내된 페이지로 보인다

## T050 분석 결과 저장 구조 설계

### 목표

분석 결과를 나중에 다시 보여줄 수 있는 최소 저장 모델을 설계한다.

### 작업 내용

- 저장 필드 정의
- 로컬 저장 또는 서버 저장 방향 결정

### 완료 조건

- 기록 페이지 확장이 가능해진다

## T051 분석 기록 페이지 구현

### 목표

저장된 분석 결과 목록을 볼 수 있게 한다.

### 작업 내용

- 카드 리스트
- 날짜/챔피언/승패 표시
- 상세 진입

### 완료 조건

- 사용자가 이전 분석 결과를 다시 열어볼 수 있다
