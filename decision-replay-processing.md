# LoL 리플레이 원본 직접 처리 여부 의사결정 문서

작성일: 2026-04-10

## 1. 목적

이 문서는 `LoL 게임 리플레이 내용을 요약하고 플레이 장단점을 분석하는 웹페이지`를 만들기 위해, `리플레이 원본(.rofl)을 직접 처리할지` 여부를 결정하기 위한 기준과 권장안을 정리한다.

이번 의사결정의 핵심 질문은 아래와 같다.

- 웹서비스의 핵심 입력을 `.rofl` 원본 파일로 둘 것인가
- 아니면 `Riot 공식 Match/Timeline 데이터` 또는 `구조화된 경기 데이터`를 기준으로 제품을 설계할 것인가

## 2. 결론 요약

최종 권장안은 아래와 같다.

- `제품 MVP`: `.rofl 직접 처리`를 핵심 전제로 잡지 않는다
- `1차 출시 방향`: `Riot 공식 Match/Timeline 기반 분석`으로 시작한다
- `병행 연구 과제`: `.rofl 서버사이드 파서 PoC`를 별도 트랙으로 운영한다

즉, 제품의 본선은 `공식 데이터 기반 웹서비스`, 연구 트랙은 `비공식 리플레이 원본 처리`로 분리하는 전략을 채택한다.

## 3. 의사결정 배경

서비스 목표는 단순히 리플레이를 여는 것이 아니라 아래 결과를 안정적으로 제공하는 것이다.

- 경기 요약
- 플레이 장점/단점 분석
- 구간별 실수와 잘한 판단 정리
- 다음 게임 개선 액션 제안

이 목표를 기준으로 보면, 우리가 필요한 것은 `리플레이 재생` 자체보다 `분석 가능한 경기 데이터의 안정적 확보`다.

`.rofl` 직접 처리는 기술적으로 가능성이 있지만, 브라우저 단독 웹서비스와의 궁합, 패치 호환성, 운영 유지보수 리스크가 크다.

## 4. 조사 결과 요약

### 4.1 공식 지원 관점

- Riot 지원 문서는 리플레이 파일이 `LoL 클라이언트를 통해 재생`되는 흐름을 설명한다.
- Riot 개발 문서의 Replay/Game Client 관련 내용은 `로컬 클라이언트 제어` 성격이 강하다.
- 브라우저에서 `.rofl`을 공식 API로 직접 파싱/분석하는 지원 모델은 확인되지 않았다.

### 4.2 기술 구현 관점

- 커뮤니티 기반 `.rofl` 파서들은 존재한다.
- 다만 일부 라이브러리는 `메타데이터 파싱` 중심이거나, `하위 호환성을 보장하지 않음`을 명시한다.
- Riot 패치에 따라 메타데이터 구조나 추출 가능 정보가 깨진 사례가 있다.

### 4.3 제품 운영 관점

- 원본 리플레이 직접 처리는 분석 정밀도를 높일 여지가 있다.
- 하지만 `비공식 포맷 의존`, `패치 취약성`, `브라우저 처리 한계`, `운영 복잡도`가 동시에 존재한다.
- 반대로 Match/Timeline 기반은 정보량이 다소 제한되지만, 웹서비스 MVP에 더 적합하다.

## 5. 선택지 비교

## 옵션 A. 브라우저에서 `.rofl` 직접 업로드 후 처리

### 설명

사용자가 웹페이지에 `.rofl` 파일을 업로드하면, 프론트엔드에서 JS/WASM 기반 파서로 직접 해석한다.

### 장점

- 사용 경험이 직관적이다
- 서버 의존도가 줄어든다
- 파일 업로드만으로 바로 분석 흐름을 만들 수 있다

### 단점

- 공식 지원 경로가 아니다
- 파서 품질과 유지보수를 직접 책임져야 한다
- 브라우저 성능과 보안 제약이 있다
- 패치 대응 비용이 높다

### 판단

- `연구용 PoC`로는 가능
- `실서비스 MVP 핵심 경로`로는 비추천

## 옵션 B. 서버에서 `.rofl` 업로드 처리

### 설명

웹은 파일 업로드만 담당하고, 서버가 `.rofl`을 파싱해 구조화 데이터를 추출한 뒤 분석한다.

### 장점

- 브라우저 제약을 줄일 수 있다
- Rust 등 고성능 파서를 붙이기 쉽다
- 분석 파이프라인을 중앙에서 관리할 수 있다

### 단점

- 여전히 비공식 포맷 역공학에 의존한다
- 패치 호환성 문제가 남는다
- 서버 비용과 운영 복잡도가 증가한다
- 실패 케이스가 생기면 사용자 경험이 흔들린다

### 판단

- `고급 기능` 또는 `실험 트랙`으로는 가치 있음
- `초기 제품 핵심 입력 방식`으로는 보류 권장

## 옵션 C. 데스크톱 헬퍼 앱 + 웹 UI

### 설명

로컬 네이티브 앱이 Riot 클라이언트/로컬 API/리플레이 환경과 상호작용하고, 웹 UI는 결과 표시와 저장을 담당한다.

### 장점

- 로컬 환경 제약을 우회하기 쉽다
- 원본 리플레이 기반 고정밀 분석에 가장 현실적이다
- 추후 영상화, 시점 제어, 장면 캡처 같은 확장성이 있다

### 단점

- 순수 웹서비스가 아니다
- 설치형 제품 경험이 필요하다
- 배포와 유지보수가 복잡하다

### 판단

- `.rofl 직접 처리`를 반드시 해야 한다면 가장 현실적
- 다만 MVP 속도를 크게 늦출 수 있다

## 옵션 D. Riot Match/Timeline 기반 분석

### 설명

Riot 공식 API 또는 이미 구조화된 경기 데이터를 받아 웹에서 요약/분석한다.

### 장점

- 공식 데이터 기반이라 안정적이다
- 웹서비스와 잘 맞는다
- 구현 속도가 빠르다
- MVP 검증에 적합하다

### 단점

- `.rofl` 수준의 세밀한 장면 정보는 제한될 수 있다
- 특정 미세 행동 분석은 어렵다
- 향후 정밀 분석을 원하면 추가 데이터 전략이 필요하다

### 판단

- `MVP 최우선 선택지`
- 현재 단계에서 가장 합리적

## 7. 최종 결정

### 결정

아래와 같이 결정한다.

- 메인 제품 경로는 `옵션 D. Riot Match/Timeline 기반 분석`
- `.rofl 직접 처리`는 `옵션 B. 서버사이드 PoC`로 별도 검증
- 브라우저 단독 `.rofl` 직접 처리는 채택하지 않음

### 결정 이유

- 우리가 먼저 검증해야 하는 것은 `분석 가치`와 `사용자 경험`이다
- `.rofl 직접 처리`는 기술적으로 매력적이지만 제품 리스크가 너무 크다
- MVP 단계에서 가장 중요한 것은 `안정적으로 입력받고, 설득력 있게 분석해 주는 경험`이다
- 공식 데이터 기반으로도 충분히 첫 제품 가설을 검증할 수 있다

## 8. 제품 전략에 미치는 영향

이번 결정에 따라 제품 구조는 아래처럼 바뀐다.

### 제품 1단계

- 입력: 소환사명, 경기 ID, 또는 구조화된 경기 데이터
- 분석: 경기 요약, 장점/단점, 라인전/중반/후반 평가, 개선 액션
- 출력: 타임라인, 핵심 이벤트, 플레이 피드백

### 제품 2단계

- 정밀 분석이 필요한 경우 `.rofl 업로드 베타` 또는 `고급 분석 모드` 검토
- 이 단계에서는 서버사이드 PoC 결과를 바탕으로 제품화 여부를 다시 평가

## 9. 권장 아키텍처

### 본선 아키텍처

- 프론트엔드: 웹 입력/결과 UI
- 백엔드: Match/Timeline 수집, 정규화, 분석 생성
- 분석 엔진: 규칙 기반 + LLM 기반 결합
- 저장소: 경기 결과와 인사이트 저장

### 연구 아키텍처

- 별도 파서 서비스가 `.rofl` 파일을 수신
- 구조화 가능한 메타데이터와 이벤트를 추출
- 추출 품질, 패치별 성공률, 유지보수 비용을 측정
- 본선 서비스와는 분리 운영

## 10. MVP 범위 제안

MVP에서는 아래만 포함한다.

- 최근 경기 1건 분석
- 경기 요약 카드
- 장점 3개
- 단점 3개
- 다음 게임 체크리스트
- 시간대별 핵심 이벤트

MVP에서 제외한다.

- `.rofl` 직접 업로드
- 리플레이 영상/시점 제어
- 자동 장면 캡처
- 패치별 원본 파서 대응

## 11. 후속 액션

### 즉시 진행

- Match/Timeline 기반 데이터 스키마 정의
- 분석 결과 UI 와이어프레임 작성
- 분석 프롬프트와 근거 이벤트 포맷 설계
- 샘플 경기 데이터로 MVP 데모 제작

### 병행 연구

- `.rofl 서버 파서` 후보 검토
- 패치별 호환성 테스트 기준 정의
- 추출 가능한 데이터 범위 문서화
- 제품 편입 조건 정의

## 12. 재검토 조건

아래 조건 중 하나를 만족하면 `.rofl 직접 처리`를 다시 주요 옵션으로 재검토한다.

- 서버사이드 파서 PoC 성공률이 충분히 높음
- 패치 변경에도 유지 가능한 운영 체계 확보
- Match/Timeline 기반으로는 핵심 사용자 가치를 충족하지 못함
- 원본 리플레이 기반 고정밀 분석이 명확한 차별화 요소로 검증됨

## 13. 의사결정 문장

현재 단계에서 우리 제품은 `.rofl 직접 처리`를 핵심 전제로 하지 않는다.  
우리는 `공식 Match/Timeline 기반 웹 MVP`를 먼저 출시하고, `.rofl 서버사이드 파서 PoC`는 별도 연구 과제로 운영한다.
