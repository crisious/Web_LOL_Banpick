## 1. 문서 목적

이 문서는 MVP 화면 구조와 정보 배치를 빠르게 고정하기 위한 와이어프레임 문서다.

목표는 아래와 같다.

- 사용자가 `입력 -> 분석 -> 복기 -> 개선 행동 이해` 흐름을 자연스럽게 따라갈 수 있게 한다
- 시각 디자인 이전에 `무엇을 어떤 순서로 보여줄지`를 결정한다
- 추후 UI 구현 시 팀이 공통 이해를 갖게 한다

## 2. 전체 사용자 흐름

1. 사용자가 랜딩 페이지에 들어온다
2. 경기 입력 또는 샘플 분석을 시작한다
3. 분석 로딩 후 결과 페이지로 이동한다
4. 경기 요약과 장단점을 확인한다
5. 다음 게임 체크리스트를 저장하거나 다시 복기한다

## 3. 화면 목록

- 화면 A: 랜딩 / 입력 페이지
- 화면 B: 분석 결과 페이지
- 화면 C: 분석 기록 페이지

MVP 우선순위는 `A -> B -> C` 순서다.

## 4. 화면 A: 랜딩 / 입력 페이지

### 화면 목적

- 제품 가치를 한 문장으로 이해시킨다
- 사용자를 가장 빠르게 분석 시작 지점으로 보낸다
- 실제 입력이 아직 어려워도 샘플 경기 체험으로 데모 가능하게 한다

### 핵심 질문

- 이 서비스가 무엇을 해주는가
- 나는 어떤 방식으로 바로 써볼 수 있는가

### 정보 우선순위

1. 제품 가치 제안
2. 입력 시작 영역
3. 샘플 분석 CTA
4. 결과 예시 미리보기

### 필수 UI 컴포넌트

- 헤더
- 히어로 섹션
- 입력 카드
- 샘플 분석 CTA 버튼
- 결과 미리보기 카드
- 주의사항/설명 영역

### 인터랙션 메모

- `샘플 경기 체험`은 입력 없이 바로 결과 페이지로 이동
- `분석 시작` 전 필수 입력 검증
- 로딩 전환은 부담 없이 짧고 명확해야 함

## 5. 화면 B: 분석 결과 페이지

### 화면 목적

- 사용자가 경기 1판을 짧은 시간 안에 이해하게 한다
- 단순 통계가 아니라 의미 있는 피드백을 전달한다
- 다음 게임에서 바꿀 행동이 무엇인지 명확히 보여준다

### 핵심 질문

- 이 경기는 어떤 흐름이었나
- 내가 잘한 점은 무엇인가
- 내가 놓친 점은 무엇인가
- 다음 게임에서 무엇을 바꾸면 좋은가

### 정보 우선순위

1. 경기 개요
2. 한줄 요약
3. 핵심 장단점
4. 시간대별 흐름
5. 체크리스트
6. 근거 이벤트

### 카드 설계 메모

#### 장점 카드

- 제목
- 설명
- 근거 이벤트
- 경기 영향

#### 단점 카드

- 제목
- 설명
- 근거 이벤트
- 개선 힌트

#### 체크리스트 항목

- 행동 문장 1개
- 왜 필요한지 짧은 설명

### 결과 페이지 UX 원칙

- 요약이 항상 맨 위에 와야 한다
- 통계는 요약을 보조해야지 중심이 되어선 안 된다
- 장점과 단점을 같은 비중으로 보여준다
- 개선 행동은 반드시 카드형이 아니라 `바로 읽히는 체크리스트형`이 더 적합하다
- 근거 이벤트는 하단에 몰아두되, 각 카드에서도 최소 1개는 직접 노출한다

## 8. 상태별 화면 고려

## 로딩 상태

사용자에게 아래를 보여준다.

- `경기 흐름 정리 중`
- `핵심 장면 추출 중`
- `장단점 분석 중`

이렇게 단계형 메시지를 주면 기다리는 경험이 덜 답답하다.

## 에러 상태

가능한 유형:

- 입력값 부족
- 경기 데이터를 불러오지 못함
- 분석 결과 생성 실패

에러 화면 원칙:

- 무엇이 실패했는지 짧게 설명
- 사용자가 다음에 무엇을 해야 하는지 명확히 제안
- 샘플 경기 체험 버튼을 항상 대안으로 제공

## 빈 상태

히스토리 페이지에서 데이터가 없을 경우:

- "아직 분석한 경기가 없습니다"
- "샘플 경기부터 시작해 보세요"

## 9. UX 세부 원칙

- 사용자는 결과 페이지에서 `3초 안에 경기 요약`을 이해해야 한다
- `장점/단점` 섹션은 스캔이 쉬워야 한다
- 긴 서술보다 `짧은 요약 + 근거` 조합이 더 중요하다
- 모바일에서는 `한 섹션 1스크린 내 이해`를 목표로 한다
- CTA는 항상 명확해야 한다
